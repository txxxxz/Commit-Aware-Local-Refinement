## 二、项目目标

本项目聚焦于离散扩散语言模型在代码生成任务中的“解码过程控制”问题。我们的核心目标不是重新训练一个更大的模型，而是研究模型在推理过程中“应该何时确定某个 token、何时继续等待、何时保留多个候选分支”，从而让代码生成更符合一种自然的生成顺序：先形成整体框架，再逐步补全局部细节。

这个目标来源于我们对 Dream 类扩散代码模型的一个直观观察：在一些生成样本中，模型会先把函数签名、控制流框架、缩进层级、括号配对等大结构逐渐稳定下来，然后再补充变量名、常量值、条件细节和局部语句。我们把这种“先框架、后细节”的生成过程称为一种自然涌现现象。相反，也存在一些不理想的情况，即模型过早地把局部细节 commit 掉，但整体结构尚未收敛，最终导致后续步骤即使继续 refinement，也很难再修复全局错误。

因此，本项目的目标可以概括为两点：

- 第一，尽可能增强这种“自然涌现”的生成趋势，让模型在结构尚不稳定时不要过早锁死局部细节。
- 第二，当模型出现“非自然涌现”倾向时，能够及时识别并干预，例如延迟提交、分配更多搜索预算，或维护多个全局候选分支，避免一开始走错之后全盘崩掉。

围绕这个目标，我们希望回答以下几个更具体的问题：

- 自然涌现现象能否被量化，而不仅仅停留在主观观察上？
- 是否可以用 entropy、attention/influence 等信号来判断“当前某个位置是否适合提交”？
- 当局部位置高风险时，是否有必要维护多个分支，而不是只保留当前单一路径？
- 在提升语法正确率和测试通过率的同时，额外推理成本是否仍然可接受？

## 三、研究动机与挑战

本项目的出发点并不是“单纯提升一个 benchmark 分数”，而是想回答一个更过程化的问题：扩散模型在代码生成时，到底有没有一种比较健康的内部演化路径？如果有，我们能不能顺着这种路径去设计 decoder，而不是每一步都急着做局部最优决策。

我们最初的观察是，Dream 在一些成功样本里会表现出明显的层次性生成特征。比如，一个函数的整体形状会先出现，像 `def`、缩进、循环框架、条件分支这些决定程序骨架的成分会先逐渐稳定；之后模型再去补全细节，比如变量名、边界条件、返回表达式甚至某些字面量。这种现象和普通文本生成里“从左到右一次性决定”的感觉不太一样，更像是全局结构逐步收敛的过程。也正因为如此，我们开始怀疑：如果 decoder 在这个过程中太早 commit 某些细节 token，是否会打断这种自然演化，让模型失去后续从全局修正局部的机会。

进一步地，我们发现代码场景对这种问题尤其敏感。自然语言里，一个词选错了，后文还有机会圆回来；但代码不是这样。一个早期 commit 的括号、冒号、缩进层级、函数签名或关键控制流，一旦错了，后面的 refinement 往往是在错误骨架上继续填东西，最后不是 parse fail，就是逻辑彻底跑偏。这说明“早早 commit”在代码生成里代价很高，而且这种代价往往不是局部的，而是会扩散到整个程序。

因此，我们开始形成一个更明确的想法：既然模型本身可能存在“自然涌现”的趋势，那么 decoder 的职责不应该只是被动读取当前 logits 的 argmax，而应该主动区分两类情况：

- 对符合自然涌现趋势的位置给予更多奖励或优先保留，也就是让结构逐步成形的过程更加稳定。
- 对不符合自然涌现趋势的位置给予更多惩罚或延迟处理，特别是那些“局部看起来很确定，但全局仍然危险”的提交。

但真正实现这件事并不容易，主要有以下几个挑战：

- 自然涌现很难直接定义。它不是一个现成标签，而更像一种生成轨迹特征。我们需要找到可观测、可计算的代理信号。
- 并不是所有高 entropy 的位置都危险。有些位置只是细节尚未确定，但不影响全局；另一些位置虽然 entropy 不高，却可能是关键结构点，一旦错了会产生严重后果。
- 代码中的风险是全局相关的。某个位置的好坏，往往取决于它与前后结构是否一致，所以只看单点分布可能不够。
- 如果为所有可疑位置都保留分支，计算量会很快爆炸，因此必须想办法只对少量真正高风险的位置启用更强的搜索。

也就是说，这个项目本质上是在解决一个“如何让 decoder 更尊重结构收敛过程”的问题。相比最终只看一个输出结果，我们更关心生成过程中哪些决策导致了成功，哪些过早提交导致了失败。

## 四、数据集描述

本项目主要采用公开代码生成数据集进行实验，重点不是训练数据规模，而是通过统一 benchmark 去比较不同 decoding 策略是否真的改变了生成过程。

- HumanEval：这是最核心的数据集。它的每条样本通常包含函数说明、函数签名以及隐藏测试，适合检验“结构正确但语义错误”和“连结构都没成型”这两类失败模式。由于它有标准测试，我们可以直接观察 decoder 调整之后，是否真的带来了 pass@1 或单元测试通过率的提升。
- MBPP：相比 HumanEval，MBPP 的任务通常更短、更直接，更适合分析 decoder 在中短代码上的行为。如果一个方法只在长样本上有效而在短样本上没有收益，就说明它可能过于依赖复杂搜索；反之，如果短样本也稳定提升，说明它更像一种通用的解码改进。
- 可选扩展：如果时间允许，我们还希望把方法扩展到 APPS 或 CodeContests 这类更长、更复杂的题目上。这样可以进一步检验“先框架后细节”的自然涌现现象在长程序里是否更明显，以及多分支维护是否在长上下文中更有价值。

我们之所以优先选择这些数据集，是因为它们能够同时覆盖三种层面的评估：

- 语法层面：生成结果是否可以被 Python 解析。
- 功能层面：生成结果是否能通过单元测试。
- 过程层面：在生成 trace 中，模型是否更倾向于先稳定结构，再补充细节。

目前仓库已经支持以 JSON 格式加载 HumanEval、MBPP、APPS 和 CodeContests，因此实验实现上不需要重新搭建完整数据管线，可以把更多精力放在 decoder 本身和 trace 分析上。

## 五、潜在解决方案

我们的解决思路不是直接发明一个庞杂的新模型，而是沿着“观察到的自然涌现现象”逐步往下拆，最终形成一个由监控、奖励/惩罚、延迟提交和多分支维护组成的 decoder。

### 5.1 从现象到机制：先定义什么叫“自然”

我们目前对自然涌现的理解是：模型应当优先让全局结构收敛，再让局部细节收敛。对于代码来说，这里的“结构”包括函数定义、控制流框架、缩进层级、括号配对、语句边界、返回结构等；“细节”则包括变量名、局部常量、局部表达式和某些实现性 token。一个更自然的生成过程应该表现为：结构相关位置的状态先逐渐稳定，而不是细节先被强行锁死。

基于这个理解，我们会尝试把“自然”转化为可操作的规则：如果某个位置的提交有助于整体结构稳定，我们倾向于奖励它；如果某个位置在全局尚不稳定时过早占据了过高置信度，我们则倾向于把它视为潜在风险，而不是直接接受。

### 5.2 用 entropy 和其他物理量监控生成状态

最直接的监控量是 entropy。高 entropy 往往意味着模型对当前位置尚不确定，因此它天然适合作为“是否应该立即 commit”的第一层判断。但仅凭 entropy 还不够，因为有些位置虽然 entropy 不高，却可能对整体结构影响很大。

因此，我们还计划引入其他辅助信号，作为“非自然涌现”的监控代理，例如：

- attention / influence proxy：衡量某个未提交位置与已稳定结构之间的耦合程度。如果一个位置与大量已提交结构强相关，那么它的错误可能会造成更大破坏。
- 局部概率间隔或 margin：如果 top-1 和 top-2 非常接近，说明当前位置虽然看似有主选项，但其实并不稳定。
- 时序信息：同一个位置在不同 diffusion step 上是否快速反复变化，或者是否长期不稳定。
- 结构一致性信号：当前位置的候选 token 是否更容易形成可解析代码，或者是否会引入明显的结构断裂。

这些信号最终会被融合成一个风险分数，用来判断当前 token 到底应该“直接提交”“继续等待”还是“触发更强搜索”。

### 5.3 对非自然涌现增加惩罚，对自然涌现增加奖励

这是整个项目最关键的设计思想之一。我们并不打算简单地把 decoder 做成一个“看到高 entropy 就停”的规则系统，而是希望它具有明确偏好：偏好那些符合结构先行趋势的候选，抑制那些破坏全局收敛节奏的候选。

具体来说，我们会从两个方向做：

- 奖励自然涌现：如果某个候选有助于提升结构完整性、语法可解析性、全局一致性，或者在多个分支中都表现出更强稳定性，那么它会得到更高评分。
- 惩罚非自然涌现：如果某个候选让局部细节过早固定，却导致结构风险上升、后续可修复性下降，或者使不同 step 间的状态剧烈震荡，那么它应该被降低优先级。

这种奖励/惩罚不一定表现为显式训练 loss，更可能体现在解码时的 branch scoring、particle scoring 或 commit policy 中。换句话说，我们是在推理阶段人为塑造一种更合理的“演化偏好”。

### 5.4 延迟 commit：不给错误骨架过早上锁

如果我们已经认为某些位置处在高风险状态，那么最直接的做法就是不要立刻 commit。为此，我们计划引入 delayed commit 机制：

- 对低风险位置，允许快速提交，减少不必要的等待。
- 对中风险位置，优先延迟一段时间，继续观察后续 step 是否让其自然收敛。
- 对高风险位置，不仅延迟提交，还可能触发局部搜索或保留多个备选分支。

这个机制的直觉非常简单：一旦错误结构被过早写死，后面的 refinement 往往只是“在错误框架里优化细节”；与其这样，不如给模型多几个 step 去完成全局协调。

### 5.5 维护多个分支，防止早期单一路径锁死全局

仅仅延迟还不够，因为有时模型面对的是多个彼此竞争的全局结构，而不是一个等待几步就会自然清晰的局部抖动。如果 decoder 只保留单一路径，那么即使它暂时不 commit，也可能隐含地朝某一个方向坍缩。

因此，我们计划在真正高风险的位置维护多个候选分支，或者使用局部粒子滤波的方法，对 top-k 候选进行短视野搜索。这样做的目的不是暴力穷举，而是在最关键的少数位置上保留“全局后悔药”。

维护多分支有两个好处：

- 可以防止某个早期局部决定过快主导整个序列。
- 可以利用后续结构信号反过来筛选更合理的分支，而不是只依据当前一步 logits 做判断。

### 5.6 局部 PF 而不是全局搜索

为了控制开销，我们不会对所有位置都进行全局 beam search，而是只在高风险点触发局部 PF。每个粒子在一个较短 horizon 内向前展开，通过语法检查、重复惩罚、稳定性和整体得分来决定哪个分支更值得保留。

这种设计本质上是在回答一个工程问题：既然不是所有 token 都重要，那就把算力花在最可能引起全局连锁错误的位置上。

### 5.7 消融实验与可解释分析

为了验证每个模块是否真的有用，我们会设计一系列消融实验，例如：

- 不使用 PF，只做风险监控和延迟提交。
- 只用 entropy，不使用 influence 或其他结构代理。
- 使用 PF，但不延迟 commit。
- 延迟 commit，但只保留单分支。
- 完整方法与原始 Dream baseline 对比。

通过这些实验，我们希望看清楚：真正带来提升的是“看到了风险”，还是“保留了多个全局可能性”，抑或是“把 commit 的时机后移”本身。

## 六、实验设计与评价方案

实验部分我们不只想证明“分数更高”，还想证明“这个方法为什么有效”。因此，实验将同时覆盖结果评估和过程评估。

### 6.1 对比方案

我们计划至少比较以下几种设置：

- Dream 或其他离散扩散模型的原生 baseline decoding。
- 仅加入 entropy 风险监控，但不做多分支。
- 风险监控 + delayed commit，但不使用 PF。
- 风险监控 + PF，但不延迟 commit。
- 只看单一物理量与融合多种信号的对比。
- 完整方法：risk-aware delayed commit + local particle filter。

如果时间允许，我们还会比较不同 risk fusion 策略、不同 commit 阈值、不同 PF 粒子数与 horizon 长度，观察方法对超参数是否敏感。

### 6.2 结果指标

最终输出层面的指标包括：

- format success rate：输出是否仍然保持为有效代码文本。
- parse success rate：输出能否通过 Python 语法解析。
- syntax error rate：语法错误比例。
- unit test pass rate 或 pass@1：在 HumanEval 等数据集上检验功能正确性。
- 平均额外前向次数：方法为获得提升所增加的计算成本。
- 平均 PF 触发次数：高风险位置是否真的只是少量触发，而不是泛滥触发。
- latency ratio vs. baseline：相对 baseline 的时间或近似前向成本比例。

### 6.3 过程指标

因为本项目强调“自然涌现过程”，所以我们还希望记录和分析一些过程性指标，而不只是最终成败：

- 哪些位置最早被 commit，它们更偏向结构 token 还是细节 token。
- 高风险位置是否集中在缩进、括号、控制流边界、返回语句等结构关键点。
- 同一个位置在多个 step 中的 entropy、risk 和 influence 如何变化。
- PF 触发后是否真的改变了最终提交结果，还是只是增加了额外计算但没有产生实质作用。
- 多分支保留下来的候选是否在后续 step 中出现明显的优胜分化。

如果这些 trace 能证明：完整方法更倾向于先稳定结构，再稳定细节，那么即使某些 benchmark 提升有限，这个项目本身也依然有研究价值，因为它揭示了扩散模型在代码生成中的一种过程规律。

### 6.4 我们预期看到的结果

我们预期完整方法相比 baseline 会呈现以下趋势：

- parse success rate 明显提高，因为错误结构被更少地过早锁死。
- unit test pass rate 有一定提升，因为更健康的结构收敛为后续功能细化提供了基础。
- PF 主要集中在少量真正高风险位置触发，而不是全局泛用。
- 计算成本会增加，但应明显低于对整个序列做大规模全局搜索的成本。

如果实验最终表明这些趋势成立，那么就可以支持我们的核心判断：对离散扩散代码模型而言，decoder 不应该只做局部贪心决策，而应该主动保护“先框架、后细节”的自然生成过程。

## 七、预期成果

本项目的预期成果不仅是一套代码实现，更是一套关于“扩散代码生成该如何解码”的解释框架。

我们希望最终得到以下几类成果：

- 一个可复现的 decoder 原型，用于在离散扩散代码模型上实现风险监控、延迟提交和局部多分支维护。
- 一组清晰的消融结果，回答 entropy、influence、delayed commit 和 PF 各自到底贡献了什么。
- 一份对自然涌现现象的经验性分析，总结哪些成功样本体现了“先框架后细节”，哪些失败样本则表现为“细节过早提交导致全局结构崩坏”。
- 一套更具解释性的 trace，可用于展示扩散模型在代码生成中的内部演化过程，而不是只给出最终输出和一个分数。

如果实验结果符合预期，那么这个项目想说明的不只是“某个 decoder 提高了几个点”，而是更进一步地说明：离散扩散模型在代码生成中可能天然具备一种结构先行的生成趋势，而一个好的 decoder 应当识别、保护并放大这种趋势，而不是用过早 commit 把它打断。
